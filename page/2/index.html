Pattern 6. Singleton模式

发表于 2017-10-02 | 分类于 设计模式学习

定义

Ensure a class has only one instance, and provide a global point of access to it.

确保一个类只有一个实例，并且使整个系统都能访问它。（提供一个全局的可以访问到它的点）

单例模式是个比较简单的模式，当你想要确保：

  • 在任何情况下都绝对只要一个实例
  • 在程序中表示出“在任何时刻只存在一个实例”

的时候，就是使用单例模式的时候

如何工作

单例模式确实比较简单，用一个简单的类来表示一下

可以看到，对于单例类来说，代码和很简单，使用private关键字修饰构造函数，保证了该类不会被外部使用new实例化（使用反射还是可以实例化的），通过getInstance方法，提供了全局的访问点。
照例，下面是单例模式的图示：

单例模式中的角色

在单例模式中，只有单例类这一个角色，该角色有一个返回唯一实例的静态方法（静态很重要）。

单例模式的优点

  1. 只要一个实例，减少开销
  2. 可以避免对资源的多重占用
  3. 可以设置全局访问点，优化资源访问

使用场景

  1. 需要定义大量的静态常量和静态方法（比如工具类，配置类）的时候
  2. 一个对象需要频繁的创建，销毁时，且创建操作很消耗资源的时候
  3. 整个项目中，只需要一个的共享访问点或共享数据（比如计数器

单例模式的缺点

  1. 对测试不友好
    不容易mock（因为没什么接口）
  2. 难以扩展
  3. 违背单一职责原则
    保证单例和业务逻辑在一个类实现。

单例模式的扩展

单例模式可以扩展成为多例模式。
只需在类里面只有一个容易保存固定数量的实例即可。

需要注意的一点

单例模式在JAVA中要注意实现方法，否则在多线程情况下，可能会产生多个实例。（饿汉和懒汉式单例

Pattern 5. Abstract Factory模式

发表于 2017-10-02 | 分类于 设计模式学习

定义

Provide an interface for creating families of related or dependent objects without specifying their concrete classes.

为创建一组相关或者相互依赖的对象提供接口，而不用指定具体类。

虽然这个定义很简单，但是我觉得这是我学前几个设计模式中，最复杂，最难理解的一个。下面进行具体的讲解。

如何工作

由于抽象工厂模式中涉及的类比较多，这个逻辑也有一点复杂，笔者就不献丑自己写实例代码了，直接结合《设计模式之禅》和《图解设计模式》之中的类图进行分析。

  1. 图解设计模式中的例子

    可以看到，整个类图分成实现类和抽象类两部分，其中，抽象类也可以分成两部分，产品类和工厂类。我们可以看到的是，产品类有多个，同时他们之间还有一定的关系（相关或者相互依赖），初步看来这个类图很符合抽象工厂模式的定义。
    那么到底是如何工作的呢？
    实际上，我们可以看到，抽象工厂类，将多个生成多个关联的对象的函数放在了一个类里。到具体的类的时候，一种产品对应一个工厂。
    实际上，我们可以看到，抽象工厂模式，本质上也就是工厂方法模式，只不过抽象工厂模式在一个类里有多个生成对象的函数（工厂方法）。
  2. 设计模式之禅的例子

    设计模式之禅中的例子，比较好的一点是，提供了两类产品（男性和女性），但原理是相同的，都是将多个相关对象的生成的操作放到了一个类中。

从以上两个例子中，我们可以总结出，抽象工厂模式本质上还是工厂方法模式，只不过将对多个对象（产品）的生成放到了一个类（工厂）中。

下面是抽象工厂模式的图示：

抽象工厂模式中的角色

  1. AbstractProduct: 抽象的产品类，用于声明产品类所持有的方法，添加对产品类的约束
  2. AbstractFactory: 抽象的工厂类，声明了如何生成多个相关对象的方法
  3. ConcreteProdcut:具体的产品类，实现了抽象产品类定义的方法，满足了约束
  4. ConcreteFactory:具体的工厂类，实现了生成对象的接口，实现了不用new，就可以生成多个相关的对象（有多个方法）

抽象工厂模式的优点

  1. 封装性
  2. 产品族内的约束非公开
  3. 类的数量比起工厂方法模式来说，大大减少

抽象工厂模式的缺点

抽象工厂模式非常难以扩展产品类，因为修改产品类，会修改大量原有类，这样违背了开闭原则，不是一个很好的设计。

应用场景

一个对象族（或是一组没有任何关系的对象）都有相同的约束，则可以使用抽象工厂模式。

与工厂方法模式的区别

对比与工厂方法模式，不同于它对每个产品都一个工厂类，抽象工厂模式把多个相关的对象的生成放到一个类中。

为什么要使用抽象工厂模式

其实在写第二节的（如何工作）的时候，我看着类图，就有一个疑问，为什么要使用抽象工厂模式，使用工厂方法模式也可以啊，而且也很完美，把生成多个对象的工作放到一个类中，也没有什么必然的联系，其中的函数也没有互相的依赖。
我查了一些资料，看到知乎上一个回答解决了我的问题，如果使用工厂方法模式，大量的产品就会带来大量的工厂类，这种设计不是很好，会使代码很复杂，抽象工厂模式，减少了工厂类的数量，把相关的或者相互依赖的类用一个工厂来生成。
那么为什么是相关的类放到一个工厂类里面呢？我猜是因为相关的类往往在一起实例化，这样可以直接实例化一个工厂类就可以了，免得又出现实例化大量工厂类的痛苦。
但是上面的解释并没有解决什么刚需（痛点），继续思考，如果是相互依赖，或者有某些关系，某些约束的类（产品），需要实例化，那么在工厂类中，很容易实现某种约束，或者某种关系。那么也就是，把相关的对象放到一个工厂类中实现，便于实现依赖。

这样一想，抽象工厂还是必要的。

Pattern 4. Factory Method模式

发表于 2017-10-01 | 分类于 设计模式学习

定义

Define an interface for creating an object,but let subclasses decide which class to instantiate.Factory Method lets a class defer instantiation to subclasses.

定义一个用于生成对象的接口，但是由其子类决定实例化哪个类。工厂方法模式使实例化工作从父类推迟到子类执行。

如何工作

在我看来，工厂方法模式实际上是替代了new关键字的工作，通过使接口的一个方法作为生产对象的工厂，来代替new关键字生成对象。
在模板方法模式中，主要有两类角色，一种是工厂，一种是产品，工厂用来生成产品。
下面还是用一段代码来描述工厂方法模式的原理吧：

可以看到，我们的工厂类和产品类是对应的，一般来说一个工厂对应一个产品，实际上从编写代码的角度上来看待问题，比如我们定义的抽象类，抽象的工厂类要知道他生产的产品，因此我们要先定义抽象的产品类，要不然的话，返回类型没有办法写，如果写Object(JAVA中的所有对象的最终父类），那就太宽泛了，不够好，这样我们生成抽象的产品类，还可以在产品类中加入对产品的约束。
我们还要注意到，由于工厂方法模式是将一个方法作为工厂（为与以后的抽象工厂模式区别），因次，该方法是整个模式的核心，类似的模板方法模式也是这样，因此，在工厂方法中，可以应用模板方法模式。

工厂方法模式中的角色

  1. AbstractProduct: 抽象的产品类，用于声明产品类所持有的方法，添加对产品类的约束
  2. AbstractFactory: 抽象的工厂类，声明了如何生成对象的方法
  3. ConcreteProdcut:具体的产品类，实现了抽象产品类定义的方法，满足了约束
  4. ConcreteFactory:具体的工厂类，实现了生成对象的接口，实现了不用new，就可以生成一个对象

为什么使用工厂方法模式

工厂方法模式实现了工厂类对具体的子类的解耦，实现了依赖倒置的设计原则，工厂类不必知道要生产的具体是什么产品，也不必知道到底是哪个子类如何实现的工厂方法，他只要知道自己生成的产品类的对象，自己有一个工厂方法即可。
工厂方法有以下几个优点：

  1. 良好的封装性
  2. 优秀的扩展性
    很容易就可以添加新的产品，而不用对原有代码做什么大的修改
  3. 屏蔽了产品类

工厂方法模式的应用场景

  1. 用于代替new创建对象，所有可以用new的地方都可以用工厂方法模式，但要考虑复杂度和实用性
  2. 构建灵活的可扩展的框架

工厂方法模式的扩展

  1. 简单工厂模式
    简单的说就是去掉抽象的工厂类，将工厂方法设置成静态的方法。这种方法简化了类图，也简化了调用方的写法
  2. 多工厂方法
    这个扩展实际上就是每个产品一个工厂类，也就相当于多个工厂类了。
    实际上，多个产品存在的话，可以只要一个产品类，只要提供要生成哪个产品的信息就可以了。但是多工厂方法，就是一个产品一个工厂类，这样不用给工厂方法传递要实现那个产品的参数，直接调用相应的生成类就可以了

Pattern 3. Template Method模式

发表于 2017-10-01 | 分类于 设计模式学习

定义

Define the skeleton of an algorithm in an operation,deferring some steps to subclasses.Template Method lets subclasses redefine certain steps of an algorithm without changing the algorithm’s structure.

定义一个算法的骨架，推迟一些步骤到子类实现。模板方法模式可以在不修改算法结构的同时在在子类重新定义一个算法的某些步骤。
在我看来，这个模式就是把不变的公共部分封装，把变化的部分抽象，同时把变化部分的实现交给子类来完成的模式。

如何工作

模板方法模式，看起来还是比较简单的，说起来还比较拗口，不太好描述，下面看一下简单的代码示例：

可以看到，在模板方法模式中，重要的是在抽象的父类中定义出模板方法，模板方法中定义了算法的框架（其实这也就是模板方法名字的来源），然后在子类中具体实现在父类中定义的抽象方法（算法的某几步），这也就是所谓的将具体实现推迟到子类完成
如下是模板方法模式的图示：

模板方法模式中的角色

  1. AbstractClass: 抽象类负责定义实现模板方法，同时定义要推迟到子类实现的抽象方法
  2. ConcreteClass:具体类，负责实现抽象父类定义的抽象方法。

模板方法模式的优点

  1. 封装不变部分，扩展变化的部分
  2. 提取公共代码，便于维护
    如果想要修改公共代码的话，直接在父类修改即可，只用改一处，便于维护，显然比复制公共代码到各个子类的方法便于维护。
  3. 行为由父类控制，子类实现

模板方法模式的应用场景

  1. 多个子类有共同的方法，同时逻辑相同
  2. 重要的，复杂的算法，可以把核心的部分作为模板方法，其他的细节推迟到子类实现
  3. 重构时，使用模板方法模式，把相同的代码提取出来，实现为父类，其他不同的地方，再在各个子类实现

模板方法模式的扩展

在《设计模式之禅》一书中，作者提出了一个模式扩展的例子。值得一提。
所谓的模板方法的扩展就是在模板方法中加入一个叫做钩子函数的东西，通过子类不同的实现方法，可以实现改变父类行为的目的。

Pattern 2. Adapter模式

发表于 2017-10-01 | 分类于 设计模式学习

定义

Adapter模式，也就是适配器模式，也有人把它称为wrapper模式。
其定义为：

Convert the interface of a class into another interface clients expect.Adapter lets classes work together that couldn’t otherwise because of incompatible interfaces.

实际上就是把现有的一个类的接口转换成另一个客户端想要的接口。使两个不兼容的接口能够一起工作。

如何工作

根据定义，我们可以看出，实际在适配器器模式中，主要涉及两个接口（类），因为，我们实现的适配器类就要能够同时访问这两个接口（类），一般来说，对于要实现的目标接口，我们的适配器类简单的声明实现这个接口就好，但是对于已经存在的另一个接口（类），我们访问他的方式就有两种了，而这两种，也适配器模式也正因为分为两种：

  • 类适配器模式（使用继承的适配器）
  • 对象适配器模式（使用委托的适配器）

适配器模式中的角色简介

可以看到，无论是类适配器的图示还是对象适配器的图示，除了client外都包含三个角色，他们分别是：

  • Adaptee:被适配的类，也就是在已经存在的旧的类，是我们要转换的类
  • Adapter:适配器类，由它来实现将旧的类转换成新的接口的操作
  • Target:目标类，定义了我们要转换到的接口

适配器模式的优点

  1. 使两个没有关系的类在一起使用
  2. 增加了类的透明性
    我们在使用的时候，只知道新的类，不知道其实工作主要是旧的类在做
  3. 增加了类的复用性
  4. 灵活性好
    当我们不需要适配器了，直接删掉就好，其他的代码都不用修改，十分灵活

为什么要用适配器模式

可以说，适配器模式是一种补救模式，是为了解决接口之间的不相容问题，如果在一开始，整个系统被设计的很好，是不应该使用这个模式的，但是当开发到一半的时候，需求有了一些改动，那么使用适配器模式，是一种低成本的方法，来实现新的需求。

Pattern 1. Iterator模式

发表于 2017-10-01 | 分类于 设计模式学习

Iterator模式定义

Provide a way to access the elements of an aggregate object sequentially without exposing its underlying representation.

Iterator模式的图示

其中各个部分的所代表的内容如下：

  1. Aggregate: 代表集合的抽象类，负责提供创建Iterator的接口
  2. Concrete Aggregate: 负责实现Aggregate所定义的接口。它可以创建具体的Iterator。
  3. Iterator:代表迭代器的抽象类，定义相关的API
  4. Concrete Iterator:实现具体的迭代器的接口

为什么要使用迭代器模式

为什么要使用迭代器模式呢？一般直接用for循环就可以搞定，用迭代器模式感觉反而变得麻烦起来了。
就我看来，迭代器模式实际上实现了迭代集合的代码对集合类的解耦，听起来有点不可思议，但确实是这样的，可以这样想，假如我们直接用for循环对某个以数组作为底层存储结构的集合类进行遍历，一旦集合类底层的存储结构换了，不能使用for循环了，那么我们就要对遍历的代码进行修改，但是如果我们使用Iterator模式，底层的数据结构变化，我们一点不care，只要在相关的Iterator实现类里面进行修改保证接口不变就好了，这样，我们的遍历代码一点都不用改，实现了遍历代码对集合类底层实现的解耦

Iterator的遍历方式

实际上，应用代码不关心Iterator是以怎样的顺序，对集合的元素进行遍历的，但一般来说，实现都是从前到后依次遍历的，但是如果对顺序没有要求的话，可以有任何顺序（比如倒序，跳格似的遍历）

在《设计模式之禅》一书中，作者称，Iterator模式已经是一种没落的模式，因为它是在是太普遍了，我们基本上不用自己实现，除了很“古董”的语言……
