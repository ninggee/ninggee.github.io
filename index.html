Linux中的搜索命令

发表于 2018-01-27 | 分类于 Linux

locate

用于搜索指定的文件

  • 调用方法 locate {file name}
  • /var/lib/mlocate 是locate查询到数据库
  • updatedb更新数据库
  • 查询遵循/etc/updatedb.conf的配置来查询
  • 配置默认不搜索一些文件夹和文件系统，来减少误解

whereis

用户查询命令以及其帮助文档的位置。

  • 调用方法 whereis {command}
  • options
    -m 只查找manual
    -b 只查找binary
  • 命令是干什么的 whatis {command}

whichis

查找命令的位置，并显示其别名，如果有的话
whichis类似于whereis

  • 对于shell内置的命令，两者都找不到，因为没有可执行文件
  • 两者都是在PATH路径里面寻找命令

find

find是Linux中最强大的搜索命令

  • 使用方法：find [搜索范围] [搜索条件]
  • find默认情况下是完全匹配的，搜寻的条件和查询结果一致，模糊查询需要使用通配符
  • 通配符
    1. * 匹配任意多个字符
    2. ? 匹配任意一个字符
    3. [] 匹配任意一个出现在括号里的字符
  • options:
    1. -name 按文件名
    2. -iname 按文件名，不区分大小写
    3. -user 按文件所有者
    4. -nouser 查找没有所有者的文件
    5. -atime -ctime -mtime {[+-]day} 查找对应时间的相应操作的文件
    6. -size {[+-]size} 查找[大于小于]正好等于size大小的文件
    7. -inum {inode} 按照inode号来查找文件
    8. {条件1} -a {条件2} 同时满足条件1和条件2
    9. {条件1} -o {条件2} 满足条件1或条件2
    10. -exec command {} \; 将find的结果替换{}然后执行command命令,\;代表一条一条结果执行command
    11. -exec command {} + 类似上一条，不过+代表全部结果执行一条命令

grep

在文件中，查找符合要求的字符串

  • 使用方法grep [options] {file name}
  • options：
    1. -v 取反，选择不包含关键字的字符串
    2. -i 忽略大小写
  • grep是包含匹配，区别于find
  • grep可以使用正则表达式来匹配
  • 匹配内容使用通配符，匹配字符串（文件内容）使用正则表达式

Pattern 22. Command模式

发表于 2017-11-02 | 分类于 设计模式学习

定义

Encapsulate a request as an object,thereby letting you parameterize clients with different requests,queue or log requests,and support undoable operations.

命令模式：将一个请求封装成一个对象，因此让你可以使用不同的请求使客户端参数化，队列化请求或记录请求日志，同时支持回滚操作。
命令模式又称动作模式或者是事务模式。

关于命令模式，其实主要是讲请求抽象化，作为变量传给客户端。事实上，是一种解耦，因为请求作为一个参数，请求的发起者不必知道处理者是谁，这样，请求的发起者，和请求的处理者就解耦了。

如何工作

我们通过看命令模式的定义，实际上就可以知道命令模式的工作原理。那就是：将一个请求封装成一个对象。看起来是十分简单的一个说法，但是只有命令的话，没有命令的接收者和发起者是没有意义的。因此，还要考虑到请求的处理者和发起者。

其中：

  • 每个命令都是一个操作，也可以说每个操作都是一个命令，针对不同的操作可能有不同的子类。
  • 命令模式使得命令的发送者，在发出命令之后，对该条命令的状态一无所知，这也正是他们的独立性的体现。
  • 命令模式关键在于有一个抽象的类或者接口表示命令类，只有实现了这个接口的类，才可以被传递，被接受者处理

命令模式中的角色

  1. receiver：接收者是命令的最终的处理者。
  2. command：抽象的command类，定义了command类必须实现的接口。
  3. concreteCommand:具体的command类，实现了定义的接口，一般来说，一个动作是一个command类。
  4. invoker：命令的调用者，实际上就是对传进来的命令调用他的处理方法。

命令模式的优点

  1. 将命令的发送和处理解耦
  2. 扩展性好，命令类非常容易扩展
  3. 容易实现命令的回滚和Redo

命令模式的缺点

  1. 命令类可能会膨胀

适用场景

  1. 需要redo和undo的场景
  2. 可以被看成是命令的场景

Pattern 21. Proxy模式

发表于 2017-11-02 | 分类于 设计模式学习

定义

Provide a surrogate or placeholder for another object to control access to it.

为一个对象提供代理或者占位符来控制对他的访问。

代理模式也被称为委托模式。

代理模式看名字就大概知道这个模式要干什么了，就是作为一个对象的代理，不过和一般的代理不同是，代理模式是作为别的对象访问被代理对象的代理，而不是我们平常的主体访问别的对象的代理。

非常有意思的一点是，对于代理模式的动机，或者作用，《图解设计模式》和《设计模式之禅》这两本书给了不同的解释，不过我觉得是侧重点的问题，《设计模式之禅》讲的全面一些，

  • 《图解设计模式》认为，代理模式推迟了被代理对象实例化的时机，使耗时任务不必在不必要的时候被代用。同时这本书也提到了，代理模式可以帮助设置访问的权限。
  • 《设计模式之禅》认为，代理模式本质是一种委托，可以提供很好的访问控制，在这本书的例子里，没有涉及推迟实例化之类的说明。代理只是简单的对被代理对象的替代。

代理模式的优点

  1. 可以控制被代理对象的使用权限
  2. 可以降低系统的耦合度，客户端不必知道真正发挥作用的到底是哪个类，只知道代理对象。
  3. 扩展性好。只要接口不变，代理类就可以被使用。

代理模式的缺点

  1. 代理类的使用了，可能导致难以理解代码

代理模式的使用场景

  1. 当一个对象的某个操作耗时特别长的时候，可以利用代理，在真正需要实力话的时候再实例化。
  2. 保护一个实例，可以控制对一个对象的访问。
  3. 可以作为缓冲，缓冲结果。

Pattern 20. Flyweight模式

发表于 2017-10-30 | 分类于 设计模式学习

定义

Use sharing to support large numbers of fine-grained objects efficiently.

使用共享来高效的生成大量细粒度对象。
由于享元模式要求共享的对象是细粒度的对象，因此享元模式又被称为轻量级模式。

享元模式只使用少量对象，这些对象很相似，状态变化很小，可以使用复用。

如何工作

要想明白如何使用享元模式，就先要理解两个概念：

  • 在享元模式中可以共享的相同内容称为内部状态（intrinsic），存储在享元对象内部并且不会随环境改变而改变
  • 需要外部环境来设置的不能共享的内容称为外部状态(extrinsic）

通过上面的概念我们可以知道，如果我们想要共享对象，肯定共享的是不会变化的那些，因为变化的那些每次都会变，肯定会创建新的。因此，我们就知道，要想实现享元模式，就一定要把不变的东西抽离出来，进行共享。下面来看一段代码：

享元模式中的角色

  2. FlyweightFactory：负责生成Flyweight的类。利用缓冲池储存可共享变量。

我理解的享元模式

享元模式主要是为了解决那种系统中可能会出现的数量巨大的只是有微小区别的对象导致的内存占用极高的问题，通过将不变的部分提取出来，并放在缓冲池共享，保证对象的数量很少。

享元模式的优点

  1. 减少内存中的对象的数量。
  2. Flyweight类中的外部状态和内部状态相互独立，不会相互响应，可以很容易得在各处被共享。

享元模式的缺点

  1. 使系统变得很复杂，逻辑实现也比较复杂。

享元模式的应用场景

  1. 一个系统中有大量的某个类的对象时。
  2. 一个类中的状态都可以外部话，可以从外部传入。
  3. 需要缓冲池的时候。

享元模式注意的问题

  1. 线程安全问题。不同的线程可能会修改同一对象。
  2. 性能问题，将内部信息单独抽象成类，可能会带来额外的性能开销。

Pattern 19. State模式

发表于 2017-10-30 | 分类于 设计模式学习

定义

Allow an object to alter its behavior when its internal state changes.The object will appear to change its class.

状态模式，允许一个对象随着他的内部状态的变化改变其行为，就好像换了一个类一样。
状态模式的核心是封装，状态的变更引起了行为的变化，从外部看起来就好像这个对象对应的类发生了改变一样。下面来看状态模式是如何工作的

如何工作

考虑状态模式是如何工作的，就要先知道为什么出现状态模式，那么为什么状态模式会出现呢？
原因很简单，是因为在项目中有很多时候，一个类的行为会随着他的状态发生变化而变化，我们正常的想法是放很多的条件判断，判断当前的状态来确定行为，但是这样的方法多了以后，就会出现大量的条件语句块。对扩展性影响极大。为了解决这类问题，状态模式就应运而生了。
状态模式的重点是通过类来描述状态，一个类代表一种状态，那么为了提取状态类。我们就要知道一个对象有哪些随状态变化而变化的行为，找到这些行为，提取他们形成单独的状态类，就是状态模式最为核心的点。下面来看示例代码吧。

我们要注意两点，是状态模式在实现的时候约定俗成的规则：

  • 把环境类的所有可能的状态列为类的静态常量
  • 环境角色有状态类的所有的随状态变化而变化的方法，在方法内使用委托来调用具体的状态类的方法。

状态模式中的角色

  1. context：环境类负责定义对外暴露的接口，这些接口基本和状态类中的方法对应，因为都是随状态变化行为不同的方法，除此之外还要提供改变环境类状态的方法。
  2. State：抽象的状态类，定义自己的抽象方法，同时保存环境变量来实现状态的转换
  3. ConcreteState: 具体的状态类,每个状态类都有两个任务,一个是在自己状态下自己特有的行为,另一个是行为发生后，利用环境类改变环境类的状态.

我理解的状态模式

我个人认为，状态模式是为了解决那些因为状态不同而行为不同，却使用条件语句来判断状态的问题，状态模式的重点是提取出那些随状态变化的方法，作为抽象类 。

状态模式的优点

  1. 结构清晰， 避免了使用大量的条件语句判断状态
  2. 将随状态变化的行为封装到一个类中，便于扩展
  3. 封装了状态转换的规则，封装性好

状态模式的缺点

  1. 可能会导致类的数量膨胀
  2. 实现结构较为复杂，看代码的时候，确实觉得比较难理解
  3. 增加一个类，可能会修改大量已有的类来添加转化规则。

状态模式的应用场景

  1. 系统中有大量随状态变化而变化的方法
  2. 系统中存在大量重复的条件判断语句

Pattern 17. Observer模式

Define a one-to-many dependency between objects so that when one object changes state,all its dependents are notified and updated automatically.

对于具体的被观察者，只要能在自己特定的时间通知观察者就可以了。
对于观察者，需要对观察对象的变化做出反应，每个观察者类的反应都不一样，因此update这个方法被设为抽象方法。

观察者模式中的角色

  1. subject：被观察者，可以动态的管理自己的观察者，同时在自己状态变化的时候，通知自己的观察者。
  2. ConcreteSubject：具体的被观察者，实现自己的逻辑，在自己觉得恰当的时间通知观察者。
  3. Observer：观察者类，观察者要在收到被观察者的通知之后，进行更新。
  4. ConcreteObserver：具体的观察者，实现自己的更新逻辑。

我理解的观察者模式

观察者模式，我个人觉得不太恰当，还是订阅-发布模式比较恰当，将那种某一线程一直观察一个对象的模式，转变成被观察的对象变化了之后，主动通知观察者，提高了效率和性能。

观察者模式的优点

  1. 观察者模式在观察者和被观察者之间建立了一种抽象耦合关系。
  2. 避免了一直观察一个对象的状态变化带来的性能开销

观察者模式的缺点

  1. 当一个被观察者有多个观察者的时候，在开发和调试的时候会比较麻烦，通知的过程会变得很长，影响性能。
  2. 当一个观察者观察多个对象的时候，可能会对某个函数返回特别慢，影响整体的性能。解决方式是异步。

观察者模式的应用场景

  1. 一个对象变化，会影响多个对象的状态，同时不知道有哪些对象会变化的时候。
  2. 编程中的一对多的情况，基本都可以利用观察者模式。

Pattern 16. Mediator模式

发表于 2017-10-29 | 分类于 设计模式学习

定义

Define an object that encapsulates how a set of objects interact.Mediator promotes loose coupling by keeping objects from referring to each other explicitly,and it lets you vary their interaction independently.

定义一个对象来封装一系列对象如何交互。中介者使对象之间不直接相互引用来减少耦合，同时使你可以独立的变化他们。中介者模式也被称为调停者模式。

中介者模式主要是用来将系统中紧密相互耦合的对象形成的网状结构转变为星型结构，减少了对象之间的相互耦合关系。下面我们来看中介者模式是如何工作的。

如何工作

对于相互紧密耦合的各个对象，通过提供一个中介者，来去除对象之间的紧密耦合，具体来说就是，原有的每个对象都存一个中介者，同时中介者也都保存各个对象，一个对象想要调用别的对象的方法，不是直接访问另一个对象，而是通过中介者，来达到对其他对象的依赖。下面还是进入我们熟悉的代码环节：

下面是中介者模式的图示：

中介者模式中的角色

  1. mediator:抽象的中介者，定义了统一的接口，用于同事类之间的通信。
  2. ConcreteMediator：具体的中介者，实现了抽象中介者的方法。
  3. Colleague： 同事类，每个同事类都知道中介者，而且通过中介者与其他角色通信，一般同事类既有自己的方法也有依赖别的同事的方法。

中介者模式的优点

  1. 减少了依赖，同事之间的交错的依赖全部转为对中介者的依赖，减少了依赖的数量。
  2. 降低了耦合。
  3. 可以简化各个同事类的实现

中介者模式的缺点

由于中介者角色包含了各个同事之间的交互逻辑，当同事类增多的时候，会变得非常臃肿，难以修改。

应用场景

  1. 系统中对象相互依赖，难以理解。
  2. 一个对象引用了大量其他对象导致难以复用。
  3. 一些交互的公共行为，希望把处理逻辑集中处理时。

中介者模式的实例

  1. MVC构架中的controller角色就运用到了中介者模式。
  2. GUI开发时，一个界面组件的复杂交互，可以把交互写到中介者中，避免处理逻辑分散到各个组件类中。
  3. MSN服务，可以利用中介者模式作为中心服务器转发聊天信息，如果没有中介者，信息就会广播到所有人的客户端，只有目标客户端才会读取信息。
