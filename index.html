Pattern 12. Composite模式

发表于 2017-10-28 | 分类于 设计模式学习

定义

Compose objects into tree structures to represent part-whole hierarchies.Composite lets clients treat individual objects and compositions of objects uniformly

将对象组织成树形结构来表示部分-整体的层次关系。组合使客户端统一的看待单个对象和组合对象。

组合模式又称合成模式，有时也被称为部分-整体模式，描述了部分与整体的关系。

可以说，凡是形成了树形结构的对象（们），都可以利用组合模式。下面来看一下组合模式是如何工作的。

如何工作

Pattern 11. Strategy模式

发表于 2017-10-27 | 分类于 设计模式学习

定义

Define a family of algorithms,encapsulate each one,and make them interchangeable.

定义一组算法，封装他们，同时使他们可以相互替换

策略模式听起来比较高大上，但是实际上来说，是一个比较简单的模式。可以说就是对封装和多态的综合应用。下面来看策略模式是如何工作的。

如何工作

根据策略模式的定义我们可以看出，策略模式的重点是封装和可替换，我们就可以肯直接的想到抽象出一个通用的接口，来面向接口进行编程，这样就实现了可替换，如果还有一些数据需要封装，那就用抽象类来封装。这也正式策略模式所使用的方法。

但是策略模式还是有一点需要注意的，那就是在策略模式中有一个Context角色，这个角色用来隔离客户端对具体策略的访问，同时封装了可能的变化。说了这么多，感觉都有点晕了，还是看一段代码比较直接。

可以看到，整个策略模式的表示代码是比较简单的，策略模式也确实是一个比较简单的模式。
面向对象编程的重点就是抽象和封装，提取共同点作为抽象父类（接口），在子类中各自体现自己的不同。对于一项工作又不同的方法去解决他，那么共同点就是解决这个问题，不同就是方法不同，我们可以看到，策略类就是这样。
关于context类，这个类比较有意思，它将策略和客户端代码隔离，这样客户端不关心具体用的什么策略，只要问题解决就可以了，同时，在解决客户端的需求时，可能只有一部分是有多种策略的，这样，其他部分可以放到 context类中，这样也算是将变化封装在这个类中。
下面是策略模式的图示：

策略模式中的角色

  1. strategy: 策略和算法的抽象类或者接口，定义了算法提供的功能。
  2. ConcreteStrategy: 具体的策略类，实现了strategy定义的接口
  3. context:隔离了客户端与策略族，也可以封装一些可能的变化

我理解的策略模式

策略模式整体来说是个比较简单的模式，看来是策略类比价重要，但是在我看来，context类才是这个模式的灵魂。
对于策略类来说，只是抽象了公共的接口，同时使用继承和多态的特性实现了策略的替换，这个对于有多种可能的策略，我们是横容易想到使用抽象和继承等方式来实现的。
但是对于context类，它是一个比较重要的类，

  • 一方面它隔离了客户端与策略类，这要求它了解有哪些策略类，同时了解客户端所需要的功能
  • 另一方面，当有一些变化的时候，比如除了策略类提供的接口还需要一些其他的操作才能完成最终的需求的时候，就需要在context类中进行一些操作了

因此在策略模式中，context类是最为关键的

策略模式的优点

  1. 可以灵活的切换算法，而不需要修改原有系统
  2. 扩展性好，添加一个策略不会对原有的系统带来什么修改
  3. 可以避免使用多种条件判断，减少了系统的复杂性

策略模式的缺点

  1. context类必须知道所有的策略类，违背迪米亚特法则
  2. 会产生很多的策略类，增加了类的数量

策略模式的使用场景

  1. 多个对象只是在行为上有不同，可以使用策略模式动态的选择哪个行为
  2. 程序要在多个算法之间切换
  3. 不希望暴露算法细节，可以使用策略类封装算法

装饰模式中的角色

  1. component: component是这个模式中的核心组件，一般为抽象类或者接口，定义了这个模式中对外开放的接口，也就是装饰器和一般对象拥有的相同的接口
  2. ConcreteComponent: ConcreteComponent是component的实现类，实现了component定义的功能。
  3. Decorator: Decorator一般为抽象类，存储了component对象，这也就是Decorator知道他要装饰的对象， 同时它也提供了和component类相同的接口
  4. ConcreteDecorator: 就是decrator的实现类

我理解的装饰模式

还是在读图说设计模式之装饰模式一节有感，书中说：

一般有两种方式可以实现给一个类或对象增加行为：

  • 继承机制，使用继承机制是给现有类添加功能的一种有效途径，通过继承一个现有类可以使得子类在拥有自身方法的同时还拥有父类的方法。但是这种方法是静态的，用户不能控制增加行为的方式和时机。
  • 关联机制，即将一个类的对象嵌入另一个对象中，由另一个对象来决定是否调用嵌入对象的行为以便扩展自己的行为，我们称这个嵌入的对象为装饰器(Decorator)

实际上，装饰器模式对应了现实中的一种需求，想要在不改变某个接口的时候，可以动态的给一个对象添加功能，要么是通过继承，生成子类的方法，另一种就是通过在对象外面包一层，同时不改变原来的接口，这样就可以通过包不同的装饰器，来动态的添加不同的功能。这就是装饰器模式的设计思想。

装饰模式的优点

  1. 灵活性
    通过不同的组合方式，可以很容易的组合出不同的功能。
  2. 低耦合（独立性好
    被装饰的对象和装饰器相互独立，可以互相不知道彼此，耦合性低
  3. 方便
    装饰器可以动态的为对象添加功能，非常的方便

装饰模式的缺点

  1. 会产生很多小的零碎的类
  2. 由于装饰器是一层包一层的，debug的时候会比较繁琐

装饰模式的应用场景

在JAVA中的InputStream和OutputStream就运用了装饰器模式

  1. 想要动态的给一个对象添加各种不同的功能的时候
  2. 不想用继承的方式为对象添加功能的时候（这么一看，装饰模式也算是继承的替代

Git中HEAD^与HEAD~的区别

发表于 2017-10-19 | 分类于 Git学习

在学习Git的过程中，有点^和~这两个符号有点迷茫。通过在StackOverFlow上查找答案，终于明白了两者的区别：

  1. ^：是回退到父元素，当父元素有多个的时候，^{n}指定第几个父亲，比如HEAD^等价于HEAD^1指向第一个父提交，同理HEAD^2指向第二个
  2. ~:指的是在继承层次上的回退，比如HEAD~2指定的是在继承层级上回退两层

Pattern 9. Bridge模式

发表于 2017-10-19 | 分类于 设计模式学习

定义

Decouple an abstraction from its implementation so that the two can vary independently.

桥接模式就是将抽象和实现解耦，使两者可以自由的变化而不相互影响

如何工作

桥接模式，也就是桥梁模式，通过组合的方式，使抽象和实现解耦。具体的来说，就是通过将继承关系转为关联关系，从而降低了类与类之间的耦合。减少了代码编写量。

下面是桥梁模式的图示：

桥梁模式中的角色

  1. Abstraction: 抽象类，该类是抽象层次的最顶层，它存储了实现类的实例。同时使用了实现类提供的基本功能，一般该类是抽象类。
  2. RefinedAbstraction: 改善的抽象类。该类位于抽象层次，一般是对父类（抽象类）的改进。在原来的类的基础上进行改进。一般该类是具体类。
  3. Impelmentor：实现者类。该类是实现层次的最顶层。定义了用于实现Abstraction角色要使用功能的接口。一般是接口或者抽象类。
  4. ConcreteImplementor: 具体的实现者。该类负责实现Implementor定义的方法。

我理解的桥梁模式

我觉得《图说设计模式》中，开头讲的一段话很好，现在分享给大家：

设想如果要绘制矩形、圆形、椭圆、正方形，我们至少需要4个形状类，但是如果绘制的图形需要具有不同的颜色，如红色、绿色、蓝色等，此时至少有如下两种设计方案：

  • 第一种设计方案是为每一种形状都提供一套各种颜色的版本。
  • 第二种设计方案是根据实际需要对形状和颜色进行组合

对于有两个变化维度（即两个变化的原因）的系统，采用方案二来进行设计系统中类的个数更少，且系统扩展更为方便。设计方案二即是桥接模式的应用。桥接模式将继承关系转换为关联关系，从而降低了类与类之间的耦合，减少了代码编写量。

上面的话就是桥梁模式的动机，当系统出现了两个变化维度时，如果我们只使用继承的方法，就会产生大量的子类，如上面的例子，就要有3*4=12个实现类，但如果使用第二种方法我们仅显示7个类就可以了，之后就可以随意组合，极大的减少了类的数量。
实际上，因为变化有两个维度，我们利用组合的方式代替了继承，就使类的变化从两个层次进行，暗暗的对应了系统变化的两个层次（一个层次对应一种维度的变化）

桥梁模式的优点

  1. 实现与抽象分离
    这没什么好说的，就是桥梁模式的定义
  2. 扩展性好
    无论是在实现层次还是在抽象层次进行扩展，都非常简单，直接在相应的层次增加实现类就可以了.
  3. 实现细节对客户端透明

桥梁模式的缺点

  1. 增加了理解系统的难度，由于聚合关系在抽象层，理解难度比较大。
  2. 桥梁模式要明确的识别出变化的两个维度，因此在使用这一模式的时候，会有一些局限性。

桥梁模式的使用场景

  1. 一个类存在两个变化的维度，且两个维度都需要扩展
  2. 不适合使用继承的场景
    实际上就是利用组合来代替继承
  3. 重用性要求比较高的场景
    实现的粒度越细，越容易被重用，使用继承，会受到父类的限制，很难出现太细的粒度。

桥梁模式和适配器模式

桥梁模式和适配器模式在某种意义上是有一些相似性的，但是这两种模式更多的是出现在软件生命中的不同阶段，桥梁模式出现在前期的设计阶段，对于两个维度的变化，可以将其分到实现层次和抽象层次。而对于适配器模式，更多的是出现在后期，系统已经实现了一部分了，发现两个部分无法协同工作，使用适配器模式。

Pattern 8. Builder模式

发表于 2017-10-02 | 分类于 设计模式学习

定义

Separate the construction of a complex object from its representation so that the same construction process can create different representations.

建造者模式：将一个复杂对象的表示和构建分离，因此相同的构建过程可以创造出不同的表示。

说实话，这个模式和模板方法模式以及工厂方法模式有一点类似。总之都是定义好构建步骤，然后把具体的构建实现推迟到子类实现，不过他们之间还是有一些区别的。这个在之后在进一步分析。

如何工作

在学习建造者模式的时候，发现《图解设计模式》和《设计模式之禅》两本书介绍的设计模式，略微有一些区别，不过仔细阅读可以发现，实际上两本书的介绍实例本质实现是相同的，只是在一些细节或者说是应用上，有一些区别。

下面是建造者模式的图示：

建造者模式中的角色

  1. Product：产品类，是我们最后构建的结果类，这里可以运用模板方法模式。
  2. Builder: 抽象的建造者类，制定了产品的建造方法，和返回方法。
  3. Concrete Builder： 具体的建造者类。实现了抽象建造者定义的方法，并返回一个建造好的产品类。
  4. Director: 指挥者类，负责控制建造步骤的顺序。

我理解的建造者模式

通过学习《图解设计模式》和《设计模式之禅》这两本书。我发现这两本书介绍的设计模式不太一样。但是也不好说哪个对哪个错，或者两个都对，只不过不同的情况应该应用不同的方法。

  1. 《图解设计模式》
    在这本书里面介绍的例子，更多是展现，由Builder类固定了构建的步骤，但是Builder的具体实现类，对Builder中定义的方法，采用不同的实现，导致了最后结果的不同（但都是属于Product类），这里介绍的东西更多的像模板方法模式。只不过，模板方法模式中的模板只是一个方法。而建造者模式的模板是一个类。
  2. 《设计模式之禅》
    在这本书里，作者更多的倾向于展现同样的步骤，但是采用的顺序不同，导致最会产生的结果的不同，同时作者也很贴心的展示了Product可以是有不同的子类实现，同时展现了在Director类中，可以控制建造的顺序，以及对客户端程序屏蔽了构建的过程。这本书里介绍的建造者模式更多的像抽象工厂模式。不过，抽象工厂模式关注的是生成一组相关或者相互依赖的对象，好像是车间生产一堆相关的零件一样。而建造者模式是建造出一个个的成熟的产品。

建造者模式的优点

  1. 封装性
    Director类屏蔽了产品构建的细节，使客户端与产品构建过程解耦
  2. 容易扩展
    由于具体的建造者比较独立，不会互相影响，添加新的建造者十分容易，不会对已经写好的建造者产生什么影响。因此容易扩展
  3. 便于控制细节
    复杂的建造过程被分解在建造者类中，使我们对建造的过程更容易控制。

建造者模式适应场景

  1. 需要生产的产品又复杂的建造过程
  2. 多个零件，可以组成不同的结果的时候
  3. 多个步骤，不同的顺序可以产生不同的结果的时候

建造者模式的缺点

  1. 产品差异比较大时，不应该使用建造者模式（不算一个Product了）
  2. 当产品建造内部的变化很复杂的时候，会产生大量的建造者具体实现类，导致系统变得庞大复杂。

Pattern 7. Prototype模式

发表于 2017-10-02 | 分类于 设计模式学习

定义

Specify the kinds of objects to create using a prototypical instance,and create new objects by copying this prototype.

使用一个原型的实例指定要创建的对象的种类，然后通过复制这个原型来创建对象。

实际上原型模式也是一个比较简单的模式，学习到现在，简单的模式一共有三个：

  • 单例模式
  • 迭代器模式
  • 原型模式

对于原型模式来说，重要的就是创建新的对象了，可以看到，重点就是通过复制的方式生成对象。

如何工作

对于原型模式如何工作，它的定义已经讲的很清楚了，就是：使用复制来生成对象。
对于JAVA来说，复制对象是一件很简单的事情，首先，Object对象中就存在一个clone方法，其次，还有一个Cloneable接口，标识了一个对象是可以拷贝的，所以，只要我们在声明类的时候，实现这个接口，并覆写clone方法就可以实现原型模式了。

原型模式中的角色

  1. Prototype: 原型对象负责定义使用复制来生成新对象的方法（接口），同时在类中可以加入对子类的约束
  2. Concrete Prototype：实现原型定义的生成新对象的接口，同时实现约束。

原型模式的优点

  1. 性能好
    原型模式（clone方法）是对内存二进制流的复制，比直接new一个对象的性能要好的多。
  2. 不会执行构造函数
  3. 可以生成一些难以通过new生成的对象
    比如要实现对用户自己画的一个图的复制，通过new一个新对象然后再修改值得方法是很复杂，很难实现的，通过原型模式，就很容易实现这样的功能
  4. 解耦框架和生成的实例
    可以让生成实例的框架不依赖具体的类，使用事先注册的对象，对其进行复制而不是使用类名来生成对象可以实现解耦。

原型模式的应用场景

  1. 类的初始化需要消耗大量资源的时候
  2. 对性能和安全要求比较高的情况
  3. 不容易通过new生成对象的时候
  4. 多个修改者可能修改一个对象的时候，可以避免相互影响

原型模式的注意点

  1. 构造函数不会被执行
  2. 深拷贝与浅拷贝的问题
  3. final关键字影响深拷贝的问题

Pattern 6. Singleton模式

发表于 2017-10-02 | 分类于 设计模式学习

定义

Ensure a class has only one instance, and provide a global point of access to it.

确保一个类只有一个实例，并且使整个系统都能访问它。（提供一个全局的可以访问到它的点）

单例模式是个比较简单的模式，当你想要确保：

  • 在任何情况下都绝对只要一个实例
  • 在程序中表示出“在任何时刻只存在一个实例”

的时候，就是使用单例模式的时候

如何工作

单例模式中的角色

在单例模式中，只有单例类这一个角色，该角色有一个返回唯一实例的静态方法（静态很重要）。

单例模式的优点

  1. 只要一个实例，减少开销
  2. 可以避免对资源的多重占用
  3. 可以设置全局访问点，优化资源访问

使用场景

  1. 需要定义大量的静态常量和静态方法（比如工具类，配置类）的时候
  2. 一个对象需要频繁的创建，销毁时，且创建操作很消耗资源的时候
  3. 整个项目中，只需要一个的共享访问点或共享数据（比如计数器

单例模式的缺点

  1. 对测试不友好
    不容易mock（因为没什么接口）
  2. 难以扩展
  3. 违背单一职责原则
    保证单例和业务逻辑在一个类实现。

单例模式的扩展

单例模式可以扩展成为多例模式。
只需在类里面只有一个容易保存固定数量的实例即可。

Pattern 5. Abstract Factory模式

发表于 2017-10-02 | 分类于 设计模式学习

定义

Provide an interface for creating families of related or dependent objects without specifying their concrete classes.

为创建一组相关或者相互依赖的对象提供接口，而不用指定具体类。

虽然这个定义很简单，但是我觉得这是我学前几个设计模式中，最复杂，最难理解的一个。下面进行具体的讲解。

如何工作

由于抽象工厂模式中涉及的类比较多，这个逻辑也有一点复杂，笔者就不献丑自己写实例代码了，直接结合《设计模式之禅》和《图解设计模式》之中的类图进行分析。

  1. 图解设计模式中的例子

    可以看到，整个类图分成实现类和抽象类两部分，其中，抽象类也可以分成两部分，产品类和工厂类。我们可以看到的是，产品类有多个，同时他们之间还有一定的关系（相关或者相互依赖），初步看来这个类图很符合抽象工厂模式的定义。
    那么到底是如何工作的呢？
    实际上，我们可以看到，抽象工厂类，将多个生成多个关联的对象的函数放在了一个类里。到具体的类的时候，一种产品对应一个工厂。
    实际上，我们可以看到，抽象工厂模式，本质上也就是工厂方法模式，只不过抽象工厂模式在一个类里有多个生成对象的函数（工厂方法）。
  2. 设计模式之禅的例子

    设计模式之禅中的例子，比较好的一点是，提供了两类产品（男性和女性），但原理是相同的，都是将多个相关对象的生成的操作放到了一个类中。

从以上两个例子中，我们可以总结出，抽象工厂模式本质上还是工厂方法模式，只不过将对多个对象（产品）的生成放到了一个类（工厂）中。

下面是抽象工厂模式的图示：

抽象工厂模式中的角色

  1. AbstractProduct: 抽象的产品类，用于声明产品类所持有的方法，添加对产品类的约束
  2. AbstractFactory: 抽象的工厂类，声明了如何生成多个相关对象的方法
  3. ConcreteProdcut:具体的产品类，实现了抽象产品类定义的方法，满足了约束
  4. ConcreteFactory:具体的工厂类，实现了生成对象的接口，实现了不用new，就可以生成多个相关的对象（有多个方法）

抽象工厂模式的优点

  1. 封装性
  2. 产品族内的约束非公开
  3. 类的数量比起工厂方法模式来说，大大减少

抽象工厂模式的缺点

抽象工厂模式非常难以扩展产品类，因为修改产品类，会修改大量原有类，这样违背了开闭原则，不是一个很好的设计。

应用场景

一个对象族（或是一组没有任何关系的对象）都有相同的约束，则可以使用抽象工厂模式。

与工厂方法模式的区别

对比与工厂方法模式，不同于它对每个产品都一个工厂类，抽象工厂模式把多个相关的对象的生成放到一个类中。

为什么要使用抽象工厂模式

其实在写第二节的（如何工作）的时候，我看着类图，就有一个疑问，为什么要使用抽象工厂模式，使用工厂方法模式也可以啊，而且也很完美，把生成多个对象的工作放到一个类中，也没有什么必然的联系，其中的函数也没有互相的依赖。
我查了一些资料，看到知乎上一个回答解决了我的问题，如果使用工厂方法模式，大量的产品就会带来大量的工厂类，这种设计不是很好，会使代码很复杂，抽象工厂模式，减少了工厂类的数量，把相关的或者相互依赖的类用一个工厂来生成。
那么为什么是相关的类放到一个工厂类里面呢？我猜是因为相关的类往往在一起实例化，这样可以直接实例化一个工厂类就可以了，免得又出现实例化大量工厂类的痛苦。
但是上面的解释并没有解决什么刚需（痛点），继续思考，如果是相互依赖，或者有某些关系，某些约束的类（产品），需要实例化，那么在工厂类中，很容易实现某种约束，或者某种关系。那么也就是，把相关的对象放到一个工厂类中实现，便于实现依赖。

这样一想，抽象工厂还是必要的。

Pattern 4. Factory Method模式

发表于 2017-10-01 | 分类于 设计模式学习

定义

Define an interface for creating an object,but let subclasses decide which class to instantiate.Factory Method lets a class defer instantiation to subclasses.

定义一个用于生成对象的接口，但是由其子类决定实例化哪个类。工厂方法模式使实例化工作从父类推迟到子类执行。

如何工作

在我看来，工厂方法模式实际上是替代了new关键字的工作，通过使接口的一个方法作为生产对象的工厂，来代替new关键字生成对象。
在模板方法模式中，主要有两类角色，一种是工厂，一种是产品，工厂用来生成产品。

可以看到，我们的工厂类和产品类是对应的，一般来说一个工厂对应一个产品，实际上从编写代码的角度上来看待问题，比如我们定义的抽象类，抽象的工厂类要知道他生产的产品，因此我们要先定义抽象的产品类，要不然的话，返回类型没有办法写，如果写Object(JAVA中的所有对象的最终父类），那就太宽泛了，不够好，这样我们生成抽象的产品类，还可以在产品类中加入对产品的约束。
我们还要注意到，由于工厂方法模式是将一个方法作为工厂（为与以后的抽象工厂模式区别），因次，该方法是整个模式的核心，类似的模板方法模式也是这样，因此，在工厂方法中，可以应用模板方法模式。

工厂方法模式中的角色

  1. AbstractProduct: 抽象的产品类，用于声明产品类所持有的方法，添加对产品类的约束
  2. AbstractFactory: 抽象的工厂类，声明了如何生成对象的方法
  3. ConcreteProdcut:具体的产品类，实现了抽象产品类定义的方法，满足了约束
  4. ConcreteFactory:具体的工厂类，实现了生成对象的接口，实现了不用new，就可以生成一个对象

为什么使用工厂方法模式

工厂方法模式实现了工厂类对具体的子类的解耦，实现了依赖倒置的设计原则，工厂类不必知道要生产的具体是什么产品，也不必知道到底是哪个子类如何实现的工厂方法，他只要知道自己生成的产品类的对象，自己有一个工厂方法即可。
工厂方法有以下几个优点：

  1. 良好的封装性
  2. 优秀的扩展性
    很容易就可以添加新的产品，而不用对原有代码做什么大的修改
  3. 屏蔽了产品类

工厂方法模式的应用场景

  1. 用于代替new创建对象，所有可以用new的地方都可以用工厂方法模式，但要考虑复杂度和实用性
  2. 构建灵活的可扩展的框架

工厂方法模式的扩展

  1. 简单工厂模式
    简单的说就是去掉抽象的工厂类，将工厂方法设置成静态的方法。这种方法简化了类图，也简化了调用方的写法
  2. 多工厂方法
    这个扩展实际上就是每个产品一个工厂类，也就相当于多个工厂类了。
    实际上，多个产品存在的话，可以只要一个产品类，只要提供要生成哪个产品的信息就可以了。但是多工厂方法，就是一个产品一个工厂类，这样不用给工厂方法传递要实现那个产品的参数，直接调用相应的生成类就可以了
